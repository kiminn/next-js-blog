**ERD(Entity Relationsship Diagram)** 는 데이터베이스를 구축할 때 가장 기초적인 뼈대 역할을 하며,

릴레이션 간의 관계들을 정의한 것입니다.

서비스 구축 시 단단한 골조의 역할로, 가장 먼저 신경 써야 할 부분입니다.

## 프론트엔드 개발자가 ERD에 대하여 알아봐야하는 이유

1. 데이터 모델 이해
   ERD를 통해 프론트엔드 개발자는 백엔드에서 사용되는 데이터 모델을 이해할 수 있습니다. <br/>
   이는 데이터 구조와 관계를 명확하게 파악할 수 있게 해 주며, 이로 인해 더 효율적으로 데이터를 요청하고 활용할 수 있습니다.

2. 인터페이스 설계
   개선데이터 모델의 이해는 사용자 인터페이스(UI) 설계에 도움이 됩니다. <br/>
   개발자는 데이터의 관계와 중요성을 이해함으로써 사용자가 필요로 하는 정보를 보다 직관적으로 제공할 수 있는 UI를 설계할 수 있습니다.

3. 요구 사항 분석과 커뮤니케이션
   ERD를 이해함으로써 프론트엔드 개발자는 프로젝트의 요구 사항을 더 잘 분석하고, <br/>
   백엔드 개발자나 데이터베이스 관리자와 효과적으로 의사소통할 수 있습니다.

4. 데이터 흐름과 로직의 이해
   ERD를 통해 데이터가 어떻게 흐르고, 어떤 로직에 따라 처리되는지 이해할 수 있습니다. <br/>
   이는 프론트엔드에서 데이터를 처리하고 표시하는 방식에 영향을 미칩니다.

## ERD의 중요성

ERD는 시스템 요구사항을 기반으로 작성되며, 이를 기반으로 데이터베이스를 구축합니다.

데이터베이스 구축 이후에도 디버깅 또는 비즈니스 재설계가 필요한 경우 설계도의 역할을 담당하기도 합니다.

#### 🔴 비정형 데이터를 충분히 표현할 수 없다는 단점이 있습니다.

## 정규화 과정

**정규화 과정** 은 릴레이션 간의 잘못된 종속 관계로 인해 데이터베이스 이상 현상이 일어나 이를 해결하거나,
저장 공간을 효율적으로 사용하기 위해 릴레이션을 여러 개로 분리하는 과정입니다.

**데이터베이스 이상 현상** 이란 회원이 한 개의 등급을 가져야 하는데 세 개의 등급을 갖거나 삭제할 때 필요한 데이터가 같이 삭제되고,
데이터를 삽입해야 한는데 하나의 필드 값이 NULL이 되면 안 되어서 삽입하기 어려운 현상을 말합니다.

**정규형 원칙**

1. 릴레이션이지만 좀 더 좋은 구조로 만들어야 한다
2. 자료의 중복성은 감소해야 한다
3. 독립적인 관계는 별개의 릴레이션으로 표현해야 한다
4. 각각의 릴레이션은 독립적인 표현이 가능해야 한다

## 프론트엔드 개발자와 데이터베이스 정규화와 정규형

### 1. 백엔드와의 커뮤니케이션 강화

정규화에 대한 이해는 프론트엔드 개발자가 백엔드 개발자와 데이터 모델,
API 설계 및 데이터 요구 사항에 대해 보다 효과적으로 소통할 수 있게 합니다.
이는 팀 내 협업을 개선하고, 개발 프로세스를 가속화하는 데 도움이 됩니다.

### 2. 복잡한 데이터 처리 능력 향상

정규화된 데이터 모델은 종종 여러 테이블에 걸쳐 관계를 맺고 있습니다.
이러한 복잡한 데이터 구조를 이해하고 다룰 수 있게 되면, 프론트엔드에서 데이터를 처리하고 표시하는 방법이 향상됩니다.
이는 특히 대규모, 데이터 중심의 웹 애플리케이션 개발에 있어 중요한 능력입니다.

---

### 제1정규형 (1NF)

모든 필드(열)의 값이 원자값(분할 불가능한 값, atomic value)이어야 합니다.

즉, 테이블의 모든 칼럼에는 반복되는 그룹이나 배열이 없어야 하며, 각 필드는 하나의 값만을 가져야 합니다.

만약 반복 집합이 있다면 제거해야 합니다.

**목적: 데이터의 단순성과 명확성을 확보**

![](/posts/제1.png)

### 제2정규형 (2NF)

릴레이션이 1NF를 만족하며, 부분 함수의 종속성을 제거한 형태입니다.

**부분 함수의 종속성 제거란 기본키가 아닌 모든 속성이 기본키에 완전 함수 종속적인 것**

즉, 기본키가 아닌 모든 필드가 기본키 전체에 의해서만 결정되어야 하며, 기본키의 일부분에 의해서는 결정되지 않아야 합니다.

**목적: 부분적 종속성을 제거하여 데이터 중복 감소, 무결성을 향상**

![](/posts/제2.png)

### 제3정규형 (3NF)

정의: 2NF를 만족하며, 기본키가 아닌 모든 속성이 이행 함수 종속을 만족하지 않는 상태를 말합니다.

**이행 함수 종속** 이란 A->B, B->C가 성립할 때, A->C가 성립하는 관계를 말합니다.

**목적: 이행적 종속성을 제거하여 데이터 중복 감소, 무결성을 개선**

![](/posts/제3.png)

### 보이스-코드 정규형 (BCNF)

3NF를 만족하며, 모든 결정자가 후보키인 상태를 말합니다.

```
결정자: 함수 종속 관계에서 특정종속자를 결정짓는 요소
X->Y 일 때, X는 결정자, Y는 종속자이다.
```

**목적: 기본키가 아닌 결정자에 의한 이상 현상을 제거**

---

BCNF, 4NF, 5NF는 상대적으로 더 복잡하고 특수한 경우에 적용됩니다.

일반적인 애플리케이션에서는 3NF까지의 정규화가 가장 일반적으로 사용됩니다.

BCNF, 4NF, 5NF는 주로 이론적인 관점에서 논의되며, 실제 애플리케이션에서 이들 정규형을 모두 엄격하게 적용하는 경우는 드뭅니다.
